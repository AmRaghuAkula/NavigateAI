The NavigateAI Framework

The 90-Day Pilot Blueprint

Every first AI initiative should follow a four-phase sequence. The total elapsed time is 90 days — long enough to gather real evidence, short enough to maintain momentum and organisational energy.

W1
Weeks 1–2 · Design
Define the solution architecture
Map the exact workflow the AI will follow — inputs, logic, outputs, and human touchpoints. Write this down before touching the tool.
⏱ 2 weeks
Document the workflow step by step
Identify every input data source and output destination
Define where human review is required
Set your 90-day success threshold in writing
W2
Weeks 3–5 · Build & Configure
Configure the tool, connect the data
Set up the tool in a test environment using real but non-production data. No shortcuts — a proper test environment prevents production surprises.
⏱ 3 weeks
Configure tool with test data — not live data
Build the workflow exactly as documented in Phase 1
Identify and handle edge cases before moving on
Get sign-off from the data owner and AI sponsor
W3
Weeks 6–10 · Test & Validate
Test with real conditions — not ideal ones
Run the system on real data with a small user group. Measure against your pre-defined success threshold. Identify failure modes before full deployment.
⏱ 5 weeks
Test with a single team or small user group first
Measure outputs against baseline — not just user satisfaction
Log every failure, error, and unexpected output
Review results against your 90-day threshold — go or no-go
W4
Weeks 11–13 · Deploy & Stabilise
Go live with oversight — then scale
Staged deployment to production with human review for the first 30 days. Only expand access once the system has demonstrated reliability at the initial scale.
⏱ 3 weeks
Deploy to full intended user group with monitoring active
Human review of outputs for first 30 days
Document lessons learned for the next initiative
Confirm handover to Stage 7 monitoring owner
The go/no-go rule At the end of week 10, you have one decision: does the pilot meet your pre-defined threshold? If yes, deploy. If not, either fix the specific failure mode or stop. Never extend indefinitely.
Before Every Deployment Decision

The Three Deployment Gates

Before any AI system moves from test to production, it must pass three gates. Each gate has a clear condition. All three must be met — not two out of three.

🎯
Gate 1 — Performance
The system meets or exceeds the success threshold defined before the pilot began. Not "it mostly works" or "users seem happy" — the specific metric you committed to at the start.
"Does it hit our pre-defined threshold on the specific metric we said it would improve?"
🔐
Gate 2 — Safety
The system has been tested for failure modes — edge cases, bad inputs, unexpected outputs — and has a documented human escalation path for anything it cannot handle confidently.
"Have we tested what happens when something goes wrong — and does someone know what to do when it does?"
🤝
Gate 3 — Ownership
A named person owns this system in production — responsible for monitoring, escalation, and the decision to pull it if performance degrades. Not a team. One person.
"Who is responsible for this system after deployment — and do they know they own it?"

✦ Ready to start building when…
The workflow is documented step by step before any tool configuration begins
A 90-day success threshold is written down and agreed by the AI sponsor
A test environment with non-production data is set up
The pilot user group (first testers) has been identified and briefed
All three deployment gates have been defined
Stage 7 monitoring owner has been identified before deployment
✓ Common mistakes to avoid
Not testing only with perfect scenarios — real data is messier than demo data
Not deploying to everyone simultaneously — stage the rollout
Not leaving pilots open-ended — every pilot needs a go/no-go date
